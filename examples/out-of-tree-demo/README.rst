=====================
Dawn Out-of-Tree Demo
=====================

This is example for the Dawn out-of-tree extension feature. Showcases:

- A custom board (a simplified ``boards/sim``)
- Custom user IO / PROG / PROTO classes (``CIOMyDummy``, ``CProgMyDummy``,
  ``CProtoMyDummy``)
- User factory registration via strong overrides of the
  ``dawn::oot::user_*_factory()`` hooks from ``<dawn/oot.hxx>``
- Two descriptor variants - hand-written C++ and a YAML descriptor
  generated by dawnpy at build time - to exercise both descriptor
  pipelines end-to-end

The standard Dawn build does **not** reference this directory. It builds
only when you point dawnpy at the demo's defconfig.

Layout
======

::

    examples/out-of-tree-demo/
    ├── boards/sim/sim/sim/
    │   ├── CMakeLists.txt
    │   ├── Kconfig                    # sources DAWN_BOARDS_COMMON +
    │   │                              #   $(DAWN_OOT_ROOT)/external/src/{io,prog,proto}/Kconfig
    │   ├── src/
    │   └── configs/
    │       ├── nsh_user_shell/        # uses descriptors/demo.cxx
    │       └── nsh_user_shell_yaml/   # uses descriptors/demo.yaml
    ├── descriptors/
    │   ├── demo.cxx                   # hand-written C++ descriptor
    │   └── demo.yaml                  # YAML descriptor (dawnpy-generated)
    ├── dawnpy_types.py                # TypeRegistration for --types-from
    └── external/
        ├── dawn_oot.cmake             # OOT CMake entry point
        ├── dawn_oot_hooks.cxx         # strong overrides of dawn::oot::user_*() hooks
        │                              #   (factories + lifecycle); attached to
        │                              #   apps_dawn by dawn_oot.cmake
        └── src/
            ├── io/                    # CIOMyDummy + CIOMyFactory
            ├── prog/                  # CProgMyDummy + CProgMyFactory
            └── proto/                 # CProtoMyDummy + CProtoMyFactory

Building
========

Two defconfigs exercise both descriptor paths. From the upstream Dawn
repo root:

.. code-block:: console

   # C++ descriptor variant
   $ python -m dawnpy build /tmp/dawn-oot \
         examples/out-of-tree-demo/boards/sim/sim/sim/configs/nsh_user_shell \
         -e CXX=g++-14 -e CC=gcc-14
   $ /tmp/dawn-oot/nuttx

   # YAML descriptor variant (dawnpy auto-passes --types-from
   # $DAWN_OOT_ROOT/dawnpy_types.py to the YAML generator)
   $ python -m dawnpy build /tmp/dawn-oot-yaml \
         examples/out-of-tree-demo/boards/sim/sim/sim/configs/nsh_user_shell_yaml \
         -e CXX=g++-14 -e CC=gcc-14
   $ /tmp/dawn-oot-yaml/nuttx

In NSH, run ``dawn &`` to start Dawn. Boot output should include::

   dawn: CIOMyFactory: creating CIOMyDummy
   dawn: CProgMyFactory: creating CProgMyDummy
   dawn: CProtoMyFactory: creating CProtoMyDummy
   dawn: CProtoMyDummy: hello from the out-of-tree demo!

confirming that all three user factories were resolved and used.

How it hooks together
=====================

1. ``dawnpy.dawn.project.Project`` walks up from the defconfig path,
   detects ``boards/`` + ``external/`` here, sets ``is_oot=True`` and
   exports ``DAWN_OOT_ROOT=<this directory>`` into the cmake/Kconfig env
   and passes ``-DDAWN_OOT_CMAKE_FILE=<this directory>/external/dawn_oot.cmake``
   when that file exists. If ``external/apps/Kconfig`` exists, dawnpy also
   exports its absolute path as ``DAWN_EXTENSION_APPS_KCONFIG``.

2. ``dawn/cmake/dawn_oot.cmake``'s ``dawn_oot_link()`` runs from
   ``dawn/src/CMakeLists.txt``. It includes the explicit hook file from
   ``DAWN_OOT_CMAKE_FILE``,
   whose ``dawn_oot_user_link_dawn()`` hook adds the OOT IO / PROG /
   PROTO source trees to the build.

3. ``dawn/apps/dawn/CMakeLists.txt`` adds ``$DAWN_OOT_ROOT`` to the
   ``apps_dawn`` include path, then calls the
   ``dawn_oot_user_setup_app()`` hook from the same explicit hook file.

4. ``dawn_oot_hooks.cxx`` provides strong overrides for the
   ``dawn::oot::user_*_factory()`` hooks (and optionally any of the
   lifecycle hooks ``user_init`` / ``user_post_load`` / ``user_on_idle`` /
   ``user_pre_shutdown``). Because ``external/dawn_oot.cmake`` attaches
   the file to ``apps_dawn``, those strong overrides beat the weak
   defaults shipped in ``dawn/src/oot.cxx``.

5. ``CDawn`` is constructed with the user factories; the IO/PROG/PROTO
   handlers consult them first, falling back to the built-ins for any
   class IDs the user factories return ``nullptr`` for.

See `Out-of-tree customization <https://railab.github.io/dawn/customization.html>`_
and `Out-of-tree API <https://railab.github.io/dawn/api/oot.html>`_ for the
full reference.
